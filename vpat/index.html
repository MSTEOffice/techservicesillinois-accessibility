Voluntary Product Accessibility Template

The Voluntary Product Accessibility Template (VPAT) is targeted at functional conformance to the Web Content Accessibility Guidelines (WCAG) 2.0, levels A and AA. This format was chosen to provide greater conceptual understanding of WCAG 2.0 accessibility requirements than that provided by the standard VPAT 2.0 format, which merely lists requirements in a checklist form.

Note: While this VPAT form allows for an abbreviated description of product accessibility, products will be evaluated by the university for comformance with the WCAG 2.0 Success Criteria. As such, each of the twelve WCAG 2.0 Guidelines below include a list of the Success Criteria for that Guideline.

A word about accessibility

The term "accessible" has been defined by the Department of Justice (DOJ) to mean that "... a person with a disability is afforded the opportunity to acquire the same information, engage in the same interactions, and enjoy the same services as a person without a disability in an equally effective and equally integrated manner, with substantially equivalent ease of use. The person with a disability must be able to obtain the information as fully, equally and independently as a person without a disability."

Regardless of conformance with any given accessibility law or guideline, a user with a given disability must be able to accomplish tasks without undue difficulty.

VPAT Form

This form contains a checklist of twelve accessibility requirement guidelines, matching those from the WCAG 2.0 Guidelines.

For each guideline below:

  1. Designate whether the application conforms, does not conform, or if the guideline is not applicable.
  2. Briefly describe how the application conforms to the guideline. If the application does not conform to a guideline, describe how the non-conformance will be resolved or supply a rationale for why the nonconformance is acceptable.

1—Perceivable

Information and user interface components must be presentable to users in ways they can perceive.

1.1 Text Alternatives

The proposed EIT provides text descriptions for images, graphs, charts, etc. as an alternative to visual content.

To fulfill this requirement, text alternatives must be appropriately descriptive of the non-text content.

Understanding Guideline 1.1

Success Criteria for Guideline 1.1

Provide text alternatives for any non-text content so that it can be changed into other forms people need, such as large print, braille, speech, symbols or simpler language.

  • WCAG 1.1.1 – Non-text Content (Level A): Provide for text alternatives of images and other non-text content, including user interface components.
1.2 Time-Based Media

If the product contains audio or video media, it makes captions, transcripts, and audio description available to the user.

University supplied multimedia content may be provided in an accessible method.

Automated captions, such as those provided by YouTube, are not of sufficient quality. Captions must use correct grammar and punctuation and (where appropriate) identify the speaker and other audio cues.

Understanding Guideline 1.2

Success Criteria for Guideline 1.2

Provide alternatives for time-based media.

  • WCAG 1.2.1 – Prerecorded Audio-only and Video-only (Level A): Provides that prerecorded audio is available in a visible format and that silent animations are available in an audible format.
  • WCAG 1.2.2 – Captions (Prerecorded) (Level A): Provide for synchronized captioning of prerecorded video and multimedia.
  • WCAG 1.2.3 – Audio Description or Media Alternative (Prerecorded) (Level A): Provide for audio description of prerecorded video and multimedia.
  • WCAG 1.2.4 – Captions (Live) (Level AA): Provide for captioning of live video and multimedia.
  • WCAG 1.2.5 – Audio Description (Live) (Level AA): Provide for audio description of live video and multimedia.
1.3 Adaptable

The proposed EIT allows users to view and interact with content in a way that they prefer or require.

Users who are blind require the information, structure and relationships in an application page to be presented in ways that do not require vision to perceive (e.g. programmatically) and is logical without the ability to see the screen. Sighted users with low-mobility (e.g. keyboard-only interaction) require keyboard navigation that makes sense for the application.

Understanding Guideline 1.3

Success Criteria for Guideline 1.3

Create content that can be presented in different ways (for example simpler layout) without losing information or structure.

  • WCAG 1.3.1 – Information and Relationships (Level A): Provides that information, structure, and relationships conveyed visually are available to users of assistive technology.

    Provides that semantic markup be used for headings, lists, emphasized or special text, and tabular data, including the association of data cells with their headers.

  • WCAG 1.3.2 – Meaningful Sequence (Level A): Provide for a reasonable and logical reading order when using assistive technology.
  • WCAG 1.3.3 – Sensory Characteristics (Level A): Provides that instructions are not conveyed only through sound, shape, size, or visual orientation.
1.4 Distinguishable

The proposed EIT ensures the most important content is obvious to users with varying abilities.

Some users will be unable to perceive information that is presented only through a change in color, such as an error state, or if color contrast is too low. Others will have difficulty reading page content if there is multimedia content that plays automatically.

Understanding Guideline 1.4

Success Criteria for Guideline 1.4

Make it easier for users to see and hear content including separating foreground from background.

  • WCAG 1.4.1 – Use of Color (Level A): Provides that information and prompts are not conveyed only through color.

    Provides that semantic markup be used for headings, lists, emphasized or special text, and tabular data, including the association of data cells with their headers.

  • WCAG 1.4.2 – Audio Control (Level A): Provides that there is a way to stop, pause, mute, or adjust volume with audio that plays automatically.
  • WCAG 1.4.3 – Contrast (Minimum) (Level AA): Provide for specified contrast between foreground and background of text and images of text.
  • WCAG 1.4.4 – Resize Text (Level AA): Provides tor content that remains readable and functional when the font size is doubled.
  • WCAG 1.4.5 – Images of Text (Level AA): Provide for the use of text, as opposed to images of text.

2—Operable

User interface components and navigation must be operable.

2.1 Keyboard Accessible

The proposed EIT ensures that the most important tasks can be completed easily with only the keyboard (tab, arrow keys, spacebar/enter).

Users who are blind and some users with low-mobility use only the keyboard to interact with a computer. To fulfill this requirement, keyboard shortcuts and/or mouse keys cannot be the primary way that a user can complete necessary tasks using only the keyboard.

Understanding Guideline 2.1

Success Criteria for Guideline 2.1

Make all functionality available from a keyboard.

  • WCAG 2.1.1 – Keyboard (Level A): Provide for functionality when using only the keyboard interface.
  • WCAG 2.1.2 – No Keyboard Trap (Level A): Provides that the keyboard focus is not trapped when the keyboard is used for navigation.
2.2 Enough Time

The proposed EIT provides enough time for users with varying abilities to interact with content.

Some users require more time to read content and perform actions than do others. For example, it can take a user who is blind up to three times longer to complete a task than some sighted users. Others require the ability to pause or stop scrolling or auto-updating information.

Understanding Guideline 2.2

Success Criteria for Guideline 2.2

Provide users enough time to read and use content.

  • WCAG 2.2.1 – Timing Adjustable (Level A): Provide for flexible time limits.
  • WCAG 2.2.2 – Pause, Stop, Hide (Level A): Provide for user control over moving, blinking, scrolling, and information that updates automatically.
2.3 Seizures

The proposed EIT prevents inducing seizures by limiting components that flash more than three times per second.

Components that flash more than three times per second can induce seizures in some individuals.

Understanding Guideline 2.3

Success Criteria for Guideline 2.3

Do not design content in a way that is known to cause seizures.

  • WCAG 2.3.1 – Three Flashes or Below Threshold (Level A): Provides that nothing flashes more than three times per second, unless the flash is very small and does not contain too much red.
2.4 Navigable

The proposed EIT provides ways that help users who cannot use a mouse or have visual impairments to navigate, find content, and determine where they are.

To fulfill this requirement, navigation via the keyboard (i.e. tab and arrow keys) must be logical. Features such as visual focus indicators and good heading structure are required.

Understanding Guideline 2.4

Success Criteria for Guideline 2.4

Provide ways to help users navigate, find content, and determine where they are.

  • WCAG 2.4.1 – Bypass Blocks (Level A): Provide for a skip navigation link or other means to bypass repetitive content.
  • WCAG 2.4.2 – Page Title (Level A): Provide for descriptive and informative page titles.
  • WCAG 2.4.3 – Focus Order (Level A): Provide for a keyboard-oriented navigation order that is reasonable and logical. Provides that links, form elements, and other user interface controls and components have a reasonable and logical navigation order.
  • WCAG 2.4.4 – Link Purpose (In Context) (Level A): Provides that the purpose of any link is understandable from its text or context.
  • WCAG 2.4.5 – Multiple Ways (Level AA): Provide for two or more means to locate content.
  • WCAG 2.4.6 – Headings and Labels (Level AA): Provides that headings and labels are descriptive.
  • WCAG 2.4.7 – Focus Visible (Level AA): Provides that the keyboard focus is visually apparent when using the keyboard to navigate.

3—Understandable

Information and the operation of the user interface must be understandable.

3.1 Readable

The proposed EIT ensure that text content can be read by people and by assistive technologies.

In order to read text properly on the web, screen reader software needs the language of a webpage, or part of a webpage for multi-language applications to be specified in the HTML code.

Understanding Guideline 3.1

Success Criteria for Guideline 3.1

Make text content readable and understandable.

  • WCAG 3.1.1 – Language of Page (Level A): Provides that the default language of content is exposed to assistive technology.
  • WCAG 3.1.2 – Langauage of Parts (Level A): Provides that changes in language are exposed to assistive technology.
3.2 Predictable

The proposed EIT appears and operates in predictable and consistent ways.

Predictability is essential for users with cognitive disabilities or those who are blind.

Understanding Guideline 3.2

Success Criteria for Guideline 3.2

Make web pages appear and operate in predictable ways.

  • WCAG 3.2.1 – On Focus (Level A): Provides that user interface components do not initiate a change of context when receiving focus.
  • WCAG 3.2.2 – On Input (Level A): Provides that changing the setting of user interface components does not automatically cause a change of context.
  • WCAG 3.2.3 – Consistent Navigation (Level AA): Provides that repeated navigational components occur in the same relative order each time they are encountered.
  • WCAG 3.2.4 – Consistent Identification (Level AA): Provides that components having the same functionality are identified consistently.
3.3 Input Assistance
The proposed EIT assists users to avoid, identify and correct mistakes.

To fulfill this requirement, sufficient instructional cues, unambiguous error messages and other mechanisms that alert users to mistakes or help them avoid mistakes are present. Such mechanisms must be perceivable to users who have visual and/or hearing impairments.

Understanding Guideline 3.3

Success Criteria for Guideline 3.3

Help users avoid and correct mistakes.

  • WCAG 3.3.1 – Error Identification (Level A): Provides that automatically detected input errors are identified and described in text to the user.
  • WCAG 3.3.2 – Labels or Instructions (Level A): Provide for labels or instructions when content requires user input.
  • WCAG 3.3.3 – Error Suggestions (Level AA): Provides that the system makes suggestions for correction when input errors are automatically detected and suggestions are available.
  • WCAG 3.3.4 – Error Prevention (Legal, Financial, Data) (Level AA): Provides that when legal, financial, or test data can be changed or deleted the changes or deletions can be reversed, verified, or confirmed.

4—Robust

Content must be robust enough that it can be interpreted reliably by a wide variety of user agents, including assistive technologies.

4.1 Compatible

The proposed EIT works with a variety of assistive technologies.

The proposed EIT must utilize programming standards and ensure that the name, role and value of each interface element can be programmatically determined.

To fulfill this requirement, users must not be required to use a particular operating system (if applicable), web browser (with reasonable limitations on version compatibility), or assistive technology.

Understanding Guideline 4.1

Success Criteria for Guideline 4.1

Maximize compatibility with current and future user agents, including assistive technologies.

  • WCAG 4.1.1 – Parsing (Level A): Provides that significant HTML/XHTML validation and parsing errors in source code are avoided.
  • WCAG 4.1.2 – Name, Role, Value (Level A): Provides that sufficient information (including identity, operation, and state) about user interface components is available to assistive technology.

Before submitting this form, please double check that all applicable fields have been filled in and any descriptions are sufficient for third parties to understand how this application conforms or does not conform to accessibility standards.
